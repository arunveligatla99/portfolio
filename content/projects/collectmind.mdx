## Problem

Collections policy lives at an awkward intersection: the rules change weekly, the rules are written by analysts not engineers, and the rules have to be auditable down to a single account decision. A pure LLM agent is too non-deterministic; a pure rules engine is too rigid for the way analysts actually want to write rules.

## Approach

Use the LLM only at the boundary, never in the loop.

- **Compile, do not interpret.** Plain-language rules are compiled, once, into a typed LangGraph state machine. The LLM does not run on every account; it runs once per policy version.
- **Closed action registry.** The compiler can only emit nodes from a vetted registry. If a rule asks for something outside the registry, the compile fails loudly rather than the agent improvising.
- **Deterministic replay.** The state machine is a pure function of inputs and policy version. Same inputs plus same policy version always produces the same plan.
- **Audit trail in plain English.** Every decision is annotated with the rule that triggered it, rendered into a human-readable timeline for ops review.

## Stack

- **Compiler / planner:** Python, LangGraph, OpenAI for the natural-language to typed-DSL step.
- **Runtime:** FastAPI, Postgres for state, Redis for distributed locks.
- **Observability:** structured logs with rule ids, OpenTelemetry traces.

## Outcomes

- Analysts ship policy changes without engineering involvement.
- Every account action is auditable down to the rule that caused it.
- Policy regressions are caught at compile time, not runtime.

## Lessons

- The hardest part of an agentic system is not the agent. It is the registry of safe operations.
- Determinism per policy version is a far stronger guarantee than "we use temperature 0".
