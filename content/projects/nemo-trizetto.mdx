## Problem

The billing application was carrying full PCI DSS scope because card data was flowing through it on its way to the processor. Even with masking and TLS, that scope drives audit cost, compensating control work, and an outsized blast radius if anything goes wrong. The right answer was to keep card data out of the application entirely.

## Approach

- **Vendor-hosted capture.** Card capture moved to a TriZetto-hosted iframe. The app never saw the PAN, only an opaque token returned post-capture.
- **Tokenized vault.** Card-on-file flows store only tokens. Charges, refunds, voids, and saved-card reuse all operate on tokens, never raw data.
- **Idempotent reconciliation.** Every transaction has a stable client-side idempotency key. Retries, duplicate webhooks, and partial vendor outages converge to the same final state without double-charging.
- **PCI evidence trail.** Logs are deliberately card-data-free, validated by automated grep gates in CI, and the SAQ-A workstream walks through the descoping with the QSA.

## Stack

- **App:** C# / .NET, WCF and REST endpoints depending on the legacy boundary.
- **Vendor:** TriZetto Payments hosted iframe and webhook callbacks.
- **Data:** SQL Server with separate audit and operational schemas.

## Outcomes

- App-side PCI scope reduced to SAQ-A.
- Zero card data persisted in the application database; only opaque tokens.
- Reconciliation is idempotent and survives retries and partial outages.

## Lessons

- Reducing PCI scope is a code architecture problem before it is a compliance problem.
- Idempotency is non-negotiable on payment flows; designing it in is much cheaper than bolting it on after the first reconciliation incident.
- The QSA is a stakeholder, not a checkpoint. Pulling them in early shortened the audit by weeks.
